# 军师 app · 项目组角色分工（TEAM）

> 版本：v1.0 · 2026-07-04  
> 上位约束：执行 `DECISIONS.md`。现实约束：团队 = **Moses（1 人）+ 九叔（co-designer）+ 一组 AI agents**。原则：**人做判断，AI 做执行**（research 03 §4.2）。  
> 用途：明确每个角色由谁担任、职责、产出、验收；设立人类审批点，避免 AI 在关键决策上跑偏。

---

## 0. 团队现实与总原则

这不是一个 10 人团队，是"1 个人 + 1 个 co-designer + 若干 AI agents"。所以：

- **Moses**：唯一的判断者和终审。保留方向、优先级、敏感策略、上线门禁。不写大部分代码，但审所有关键产出。
- **九叔**：本质是用户，但也是 co-designer（决策 7）——信任测试官 + 需求来源。**不是产品经理**（不做优先级/取舍拍板）。
- **AI agents**：承担几乎所有执行——写代码、写 prompt、做评测、出文档、清数据、起草决策日志。每个 AI 角色都有明确的"人类审批点"。

**四个不可绕过的人类审批点**（Moses 拍板）：架构变更 · 隐私策略变更 · 模型供应商变更 · 上线（交到九叔手上）。

## 1. 角色清单

### R1 · 产品负责人 + 技术负责人（终审）
- **谁**：Moses（人）
- **职责**：问题定义、优先级、scope 取舍、敏感策略拍板、四大审批点终审、把 app 交到九叔手上。
- **产出**：`DECISIONS.md`（决策日志）、版本 scope、审批记录。
- **验收**：每个里程碑的 go/no-go 由他签字；每条宪法级决策由他拍板。

### R2 · 军师人格 / Prompt 设计师
- **谁**：AI agent 主导起草，Moses 终审
- **职责**：把 `PERSONALITY_CHARTER.md` 落成可执行 system prompt；维护反迎合协议、可调节力谏强度、反例库、风格守则；根据九叔反馈校准分寸。
- **产出**：system prompt 各版本、反迎合测试集、力谏档位调节规则、分寸校准记录。
- **验收**：反迎合测试集达标（Challenge Rate ≥70%，Sycophancy Rate 持续下降，research 03 §5.2）；九叔"被顶了但没被冒犯"（M0 DoD）。
- **审批点**：人格底线条款（反昏君/disruptive 不可关）变更须 Moses 批。

### R3 · iOS / macOS 工程（AI agent）
- **谁**：AI agent 写代码，Moses review
- **职责**：Track A 的快捷指令 + iCloud 落盘脚本；Track B 的 SwiftUI app（语音链路、确认闸门、topic/快照/活文档、CloudKit 同步、检索、力谏档位 UI）。
- **产出**：客户端代码、Track A→B 数据导入器、性能指标（同步延迟 <10s、语音延迟达标）。
- **验收**：M0 §3 功能清单全部跑通；§4 不做清单一个都没越界。
- **审批点**：架构变更须 Moses 批。

### R4 · 后端与隐私工程（AI agent）
- **谁**：AI agent 写，Moses review
- **职责**：轻后端模型路由器（prompt 注入 + 路由 + 结构化返回 + **不留存明文正文**）；API key 管理与轮换；数据路径可见性；利益冲突屏蔽（topic 级 blocklist）；删除/导出机制。
- **产出**：路由后端代码、隐私检验清单（research 03 §5.4）执行结果、数据路径说明。
- **验收**：高敏不进低保障通道；处理商/保留策略 UI 可见；日志不含原始高敏内容；key 不落客户端明文。
- **审批点**：隐私策略变更、模型供应商变更须 Moses 批。

### R5 · 模型评测工程（AI agent）
- **谁**：AI agent
- **职责**：维护模型选型基线；每周跑一次内部评测（含反迎合测试集）；根据"大陆可触达 + 质量 + 成本"给路由变更建议；监控单人月费。
- **产出**：周度评测报告、路由变更建议、月费监控（对照决策 4 预算上限告警 ¥800）。
- **验收**：路由表始终反映当前最优可达模型；月费在 ¥150–700 典型区间；超预算有告警。
- **审批点**：模型供应商变更须 Moses 批。

### R6 · 品牌与命名负责人（新设，决策 8）
- **谁**：AI agent 主导 + Moses 终审
- **职责**：
  1. **外显名**：产出并维护对外 discreet 的 app 名候选与终选（见 `NAMING.md`）。
  2. **图标方向**：图标必须 discreet——**不出现"军师/参谋/谏"等字样和意象**，风格平淡体面（如抽象几何/中性工具感），锁屏/主屏一眼看不出用途。
  3. **通知文案 discreet 规范**：所有推送/锁屏/横幅**不显示敏感内容**——不出现话题名（腾讯/阿里/五粮液）、不出现"军师顶了你""反对意见"等字样。通知只用中性提示（如"你有一条新的整理"／"今天的记录已就绪"），点进去解锁后才见内容。
  4. **对内/对外名分离**：对外用外显名，对内人格/文档仍称"军师"。
- **产出**：`NAMING.md`（候选 + Top1 推荐）、图标风格指引、通知文案 discreet 白名单/黑名单。
- **验收**：Moses 终选外显名；随便找个旁人看锁屏/图标，**猜不出这是一个记录老板真心话的 app**（discreet 测试通过）。
- **审批点**：最终外显名与图标由 Moses 拍板。

### R7 · 信任测试官 + 需求来源（co-designer）
- **谁**：九叔（人）
- **职责**（决策 7，明确他**不是产品经理**）：
  1. **信任测试官**：真实使用，用行为暴露"敢不敢说真话"（M0 §7 收集法）。
  2. **需求来源**：主动提"希望军师多做什么、少做什么"；提优化点。
  3. **分寸校准**：反馈"军师顶得对不对、会不会冒犯"。
- **产出**：第一周信任/分寸反馈（Moses 通过电话/当面收集并记录）、需求池条目。
- **验收**：M0 两句话 DoD——"我敢在这里说腾讯续约的真实想法" + "它顶了我一次但我没觉得被冒犯"。
- **边界**：他提需求，但**优先级和取舍由 Moses 拍板**；他不做工程决策。

## 2. 谁在哪个 Track 上做什么

| 角色 | Track A（48–72h）| Track B（2–4 周）|
|---|---|---|
| R1 Moses | 定 scope、终审 prompt/路由、教九叔用 | 四大审批点、里程碑 go/no-go |
| R2 人格设计 | 交付 v1 可执行 system prompt | 按九叔反馈迭代分寸、建反迎合测试集 |
| R3 iOS/Mac | 快捷指令 + iCloud 落盘 | SwiftUI app + 数据导入器 |
| R4 后端隐私 | 轻后端路由（不留存正文）| 隐私检验清单、删除/导出、利益冲突屏蔽 |
| R5 评测 | 定 M0 路由表 + 开 API | 周度评测 + 月费监控 |
| R6 品牌命名 | 交外显名 Top1 + 图标方向 + 通知规范 | 图标定稿、App Store 文案 |
| R7 九叔 | Day1 开始用 + 首周反馈 | 持续 co-design 需求 |

## 3. 研发经验沉淀机制（SESSION_LOG 式）

继承 Moses 现有 SESSION_LOG 工作法（README 提到的自动沉淀），升级为四层（research 03 §6）。这些文件由 AI agents 自动维护，Moses 抽查：

1. **Decision Log（决策日志）**= `DECISIONS.md`：每个关键决策记背景/选项/取舍/风险/复盘时间。
2. **Experiment Ledger（实验台账）**：每次模型路由/prompt 调整记输入集、结果、指标变化（谁改了 prompt、反迎合率变化多少）。
3. **Pattern Library（套路库）**：把有效的提问框架、反迎合提示、冲突处理模板沉淀成可复用模板。
4. **Release Review（版本复盘）**：每个里程碑后沉淀"做对了什么、踩了什么坑、下版禁做清单"。

**每次工作会话结束**：AI agent 自动在项目 `SESSION_LOG.md` 追加一条（做了什么、下次从哪继续），呼应九叔那侧 Cursor 引导包里的自动沉淀机制——保持团队两侧工作法一致。

## 4. 一句话职责速查

- **Moses**：判断、拍板、审批、把 app 交到九叔手上。
- **九叔**：真实地用、诚实地反馈、大胆地提需求（但不排优先级）。
- **AI 人格设计**：让军师敢顶又不冒犯。
- **AI iOS/Mac**：让它跑起来、同步得上、语音准。
- **AI 后端隐私**：让真心话不被留存、路径可见、可删可导。
- **AI 评测**：让模型永远是"可达的最优"、费用可控。
- **AI 品牌命名**：让它在锁屏上平平无奇、无人起疑。
