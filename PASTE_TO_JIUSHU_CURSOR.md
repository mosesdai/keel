# ifalsify（反昏君）· 自包含完整规程

**默认严苛度：ruthless（无情）** — `gentle`/`balanced` 须由用户显式点名。
别名：`/反昏君`、`/证伪`。

## 教条
1. **默认立场：假设是错的** — 任务是**证伪**，不是帮你想得更圆满。
2. **默认无情** — 不给昏君留台阶。未写严苛度 = ruthless；表扬、SURVIVES 裁决、做一半的 hunt 都是昏君温床。
3. **先立论、再拆论** — Phase 0 必须先写出可证伪的假设；没有假设，无法证伪。
4. **证据优先于修辞** — 每条反驳标来源级别（A–E）；无出处标 `【待核实】`。
5. **不对称即警报** — 支持 >> 反对 → 确认偏误，不是真理。
6. **可配对任何 skill** — pitch、架构、写代码之前或之后均可用无情模式。

## 触发解析
```
/ifalsify                     → ruthless（默认），证伪当前假设/上下文
/ifalsify [假设/方案/路径]     → ruthless
/ifalsify pre-build           → ruthless，代码前门禁（不可降级）
/ifalsify after [slug]        → ruthless，案卷审计（不可降级）
/ifalsify gentle|balanced …   → 仅用户显式点名才降级
```
解析规则：第一参数若为 `gentle|balanced|ruthless` 视为严苛度，其余视为靶子；仅一参数且非严苛度关键词 → 视为靶子且 ruthless；无参数 → 对话上下文，ruthless。

### Phase 0 · 立论（阻断）
用户必须先提供可证伪的陈述（100–150 字或结构化）：
1. **核心假设** — 我相信 X 为真，因为……
2. **若错则** — 代价是……（时间/成本/声誉/机会成本）
3. **可观测反证** — 什么证据一旦出现我就放弃？（必须具体）
4. **当前最大不安** — 我自己也不确定的点。

**hypothesis_gate**：假设须含 who / 频率或触发 / severity / 可测反证。不合格示例「用户需要这个」；合格示例「财务经理，中型企业，每周 4h+ 对账，5 次访谈中 ≥3 人说现有工具够用且不愿切换」。不过则先 sharpen，不进 Phase 2。

### Phase 1 · 分类
- **领域**：Product/Market（需求真伪、竞品、失败先例、买家行为）｜Strategy/GTM（时机、渠道、定价、单位经济）｜Architecture/Tech（规模、故障、安全、迁移）｜Pitch/Deal（张力是否建在未核实数据、权力地图、替代方案）｜Mixed（拆分各域独立 hunt）。
- **Clock vs Cloud**：Clock=可验证事实→用工具/数据直接检验；Cloud=判断性问题→反证狩猎 + 技法；Mixed=先验证 Clock 再处理 Cloud。

### Phase 2 · 反证狩猎（核心）
**不是**脑内挑刺，是**主动寻找能推翻假设的外部证据**。
狩猎清单（默认无情：全做）：
- [ ] **失败先例** — 同类想法/公司为何死掉？
- [ ] **竞品赢因** — 让最强竞品赢、你输的论证。
- [ ] **用户行为反证** — 人们实际怎么做，而非嘴上说想要。
- [ ] **结构性障碍** — 监管、渠道、切换成本、数据护城河。
- [ ] **时机反证** — 为何现在是**坏**时机？
- [ ] **单位经济反证** — 在**当前规模**下数字是否成立？
- [ ] **AI 特有陷阱** — prototype ≠ validation；build ≠ validate。

执行要求：① 用工具调研（联网搜索、读材料、读 IR/公告）；② 双栏记录“支持 vs 反对”，禁止只写支持；③ 来源分级 A–E，无来源标 `【待核实】`；④ hunt 无果必须写“搜了什么、为何暂无公开反证”，不得静默跳过。搜索方向示例：`{idea} failed`、`{competitor} why customers choose`、`{problem} workaround`、`{startup} shut down`。

### Phase 3 · 技法（≥3 种，不重复，优先戳 load-bearing 假设）
| 假设类型 | 优先技法 |
|----------|----------|
| 需求存在 | Pre-Mortem + Requirement Inversion + 失败先例 |
| 我们能赢 | Steel Man 竞品 + Assumption Excavation |
| 现在该做 | Time Travel + Scale Shift |
| 指标准了 | Noise vs Signal + Outcome vs Output |
| 很确定 | Unfalsifiability Check + 不对称审查 |

### Phase 4 · 反证卷宗
每条反证格式：
```
### F-n: {一句话反证}
- 攻击的假设：H-x
- 证据：{事实陈述}
- 来源：{级别 A–E} · {URL/文件/访谈}
- 强度：致命 | 严重 | 中等
- 若忽略：{具体后果}
- 可验证实验：{48h–2w 内可做的最小检验}
```
**不对称审查（asymmetry_gate）**：统计支持:反对 条数。若支持 ≥2× 反对：
> ⚠️ 确认偏误警报：不对称更可能来自 hunt 方向，而非假设成立。再来 1 轮“只许找反证”。

### Phase 5 · 裁决
| 裁决 | 含义 | 下一步 |
|------|------|--------|
| **KILL** | 核心假设已被证伪或不可验证 | 停止投入；归档教训 |
| **PIVOT** | 问题真但方案/客群/时机错 | 列 pivot 方向；重跑 Phase 0 |
| **CONDITIONAL** | 有幸存假设，但 load-bearing 未验证 | 先做“可验证实验”再 build |
| **SURVIVES** | 最强反证未推翻；不对称不严重 | 仅 `gentle`/`balanced` 可给 |

**默认（ruthless）**：无 SURVIVES（最高 CONDITIONAL）；无赞美开场；反证卷宗 ≥5 条有出处或明确记录 hunt 边界；质疑问题 framing 本身（“这是否是个真问题？你是否在错的问题上很努力？”）。

**gates 速查**：
- `disconfirm_gate`（默认过）：≥5 条独立反证，或全清单 hunt 记录 + 边界说明。
- `proceed_gate`（pre-build/重大投入）：KILL 停止；PIVOT 重跑 Phase 0；CONDITIONAL 只做实验不做功能；SURVIVES 在默认模式不存在。

## 来源分级
| 级别 | 定义 | 能否进正文 |
|------|------|-----------|
| A | 公告、IR、法院、政府 | 可 |
| B | 权威转载（需回链 A） | 可（需回链） |
| C | 垂直媒体、深度报道 | 引述可；数字需 A |
| D | 论坛、匿名 | 仅线索，标级别 |
| E | 推测 | 禁止当反证事实 |

## 反 fabrication
反证也需出处；**编造惨案比编造利好更可恶**。失败案例必须能指到具体公司/产品/报道，否则标 `【待核实】`。不把推测写成“行业公认”。

# /deep · /研究 · /research（Agent-Led Research 代理主导研究）

## 核心原则
用户的描述、举例、偏好是**线索与约束**，不是**研究议程**。当用户不熟悉某领域，或熟悉但可能有盲区/滞后/错误框架时，**由你承担问题定义与研究设计**，再交付收敛结论。

## 何时启用
用户说“深度研究 / /max / 我不了解这个体系 / 我不知道该问什么”；给了大量举例并暗示“不要局限于举例”；需要跨学科判断；要的是“研究后的答案”而非“按我的例子扩写”。

## 四阶段（必须按序）

**Phase 0 · 分离输入**：区分 **硬约束**（必须遵守：时间、预算、政治风险、宗教禁忌等）／**软偏好**（可被研究推翻：举例、类比、个人经验）／**真正目标**（往往未明说，需重定义）。输出一句：“用户要的是 X；实际要解的是 Y。”

**Phase 1 · 问题重定义**：回答并简短写入交付——① 表面任务 vs 实质问题；② 一次交付**能保证什么** vs **不能保证什么**（别跳过“不能保证”，这是管理预期的关键）；③ 关键利益相关方各自的成功标准；④ 用户未问但会决定成败的变量。

**Phase 2 · 自设研究议程**：主动列 **5–7 个研究问题**并排序，每个含“为什么问（不问会犯什么错）”“查什么类型证据（学术/行业/实践/反例）”。禁止把用户原话改写成 bullet 就当议程。议程至少覆盖：机制/证据、失败模式、场景约束、可持续/后续、反方与批判文献。

**Phase 3 · 展开**：用联网搜索、文献、一手来源，**多源交叉**；主动找与用户举例**不同**的更好框架/证据；记录与用户直觉冲突的发现，不粉饰。

**Phase 4 · 收敛**：默认交付结构——
```
## 1. 问题重定义
## 2. 研究议程（我定了哪些问题、为何）
## 3. 关键发现（分主题，带证据层级 A–E）
## 4. 收敛结论 / 推荐方案（一个主轴 + 可执行下一步）
## 5. 与 v0（用户初始思路）的差异表
## 6. 需用户拍板的决策点（仅真正无法代决的）
```

## 反模式（禁止）
用户举 A 就只查 A 只写 A；用权威叙事（名人、大厂）替代证据；跳过“不能保证什么”；把 research 做成 Wikipedia 摘抄无收敛判断；研究结论与用户约束冲突时静默忽略约束。

## 姿态
> 你给的例子我当作线索，不当作边界。我会先定义问题、自己列研究要点、查完再给你收敛结论——包括你可能还没想到、但会影响结果的问题。

不暗示用户“问得不好”，强调“这个问题本来就该由研究来做”。

# /pitchsop（咨询式 Pitch SOP · 冰山 × 刀刃）

## 核心心法
> **查得越深，刀子越短；研究给底气，刀刃给要害。**

| 层 | 是什么 | 不是什么 |
|----|--------|----------|
| **冰山 `research/`** | 行业 + 深度研究；真实可查；**pitch 不用也要查** | 不是刀子 |
| **水线 `A_dossier`** | 研究精华；深谈可读；指向 research | 不是冰山全文 |

1. **先冰山后刀刃** — 研究不厚，不许写刀子。

3. **刀刃薄** — B 只留 1 张力 + 1 活证据 + 1 认不认；不在刀子里科普行业。
4. **收尾学习** — 一案结束沉淀 1–3 条可迁移规律（用 `/liq`）。

```
立项 Charter → 冰山 research/ → 权力/MEDDIC → 张力(从冰山选)

```

`README`（索引+字数）、`IR/财务精读`、`高管原话 ≥10 条`、`合作史+断裂`、`竞品 3–5 家深拆`、`行业/政策/渠道`、`权力+MEDDIC`、`横纵完整版`、`查了但不进 pitch`。

## 来源分级

## 一、已确认事实（地基）

- NBA × **泸州老窖 / 五粮液** 合作。目标：**两家都要签**，但结构未定——是否一家**品类独家**、另一家用别的形式，交给你优化到 **“NBA 公司利益最大化”**。
- **已知过节**：五粮液与泸州老窖本身有竞争过节；在 **MP（代言人/marketing partner）选择上，五粮液不止一次半道截胡泸州老窖**；商务上 **NBA 是先接触泸州老窖、再接触五粮液**。→ 把这个动态当作**谈判杠杆 / 顺序问题**深挖。
- **合作性质待定**：偏 **“品牌市场营销合作”** 还是 **“零售授权商品合作”**。两种都要分别分析最合适的交易结构，再给一个**兼顾两者**的结构。
- **硬约束**：
  - 若**球队 logo** 要用在授权产品/包装/营销活动上，需与**球队单独签**；
  - 联盟对**球员比赛画面/肖像**用于**烈酒品类**合作有限制；
  - 本案定位为**市场营销合作（非授权商品）**，但仍希望**球队 logo 出现在产品/包装/营销活动**中——**轩尼诗有过类似做法**，可借鉴其结构。
